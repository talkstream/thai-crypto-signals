# Thai Crypto Signals — практическое руководство по сбору биржевых курсов на Cloudflare Workers

> **Репозиторий:** https://github.com/talkstream/thai-crypto-signals ·
> **Живое демо:** https://thai-crypto-signals.mommyslittlehelper.workers.dev

Небольшой production-сервис, который каждые две минуты забирает курсы криптовалют из публичного API
биржи **Bitkub**, хранит их точно и отдаёт через компактный read-API — целиком на **Cloudflare
Workers**. Он же — бесплатный разбор на живом примере: как интегрировать публичный API на
serverless-инфраструктуре с той аккуратностью, которая нужна настоящему сервису. Сделано открыто для
студентов и тех, кто учится сам.

> **Только для обучения.** Это не финансовая рекомендация и не заявление о соответствии регуляторным
> требованиям. Торговля цифровыми активами несёт высокий риск. Подробности — в разделе
> [Предупреждение о рисках](#risk-notice) ниже.

## Для кого это

Студент или начинающий разработчик, который читает базовый **JavaScript/TypeScript** и умеет
работать в терминале. Опыт с Cloudflare или блокчейном не нужен — всё остальное объясняется по ходу.

**Что нужно заранее**
- Установленные Node.js **22+**, **pnpm** и **git**. (Проверьте: `node -v`, `pnpm -v`, `git -v`.)
- Для необязательного раздела с деплоем — бесплатный аккаунт Cloudflare и немного уверенности в
  правке конфигурационного файла JSON.

**Время:** около 30 минут на чтение и локальный запуск, если инструменты уже установлены; в первый
раз дольше, пока настраиваете Node и pnpm. Раздел с деплоем добавит примерно полчаса.

## Чему вы научитесь

1. Как устроен **публичный API биржи** и как получать курсы, не нарушая его правил.
2. Как **serverless-cron** запускает код по расписанию без сервера, который надо обслуживать.
3. Почему деньги хранят в **целых числах**, а не во float, — и как это сделать.
4. Как сделать повторяемую задачу **идемпотентной** (безопасной к двойному запуску) средствами базы.
5. Как тестировать код на **реальной инфраструктуре** вместо заглушек.

### Мини-глоссарий
- **API** — заданный способ одной программе запрашивать данные у другой по сети.
- **Endpoint** — конкретный URL API, который отдаёт один вид данных (например, все текущие цены).
- **Serverless / Cloudflare Workers** — вы загружаете функцию, платформа сама запускает её по запросу
  и по расписанию. Машину обслуживать не нужно.
- **cron** — расписание запуска задачи (например, «каждые 2 минуты»).
- **D1** — встроенная SQL-база Cloudflare (SQLite).
- **KV** — быстрый key-value-кэш Cloudflare.
- **bigint** — целочисленный тип JavaScript без ограничения размера, в отличие от обычного `number`.
- **минорные единицы** — наименьшая целая доля величины (как сатанг или сатоши); хранится целым
  числом, чтобы не терять дробную часть.
- **идемпотентность** — операцию можно запустить хоть сто раз, результат будет тот же, что и от
  одного запуска; повторять безопасно.
- **временной bucket** — отметка времени, округлённая вниз до фиксированного окна (здесь — 2-минутный
  слот); по ней группируют и дедуплицируют каждый тик.
- **OHLC** — Open/High/Low/Close, четыре числа, описывающие цену за окно (свеча).

## Контекст: из чего всё состоит

**API биржи.** У Bitkub есть бесплатный публичный read-only API. Здесь важны три endpoint'а, и ни
один не требует ключа:
- `GET /api/v3/market/symbols` — каталог торгуемых рынков (например, `BTC_THB`) и **price scale**
  каждого рынка (сколько знаков после запятой в цене).
- `GET /api/v3/market/ticker` — текущий снимок по **всем** рынкам **одним** запросом.
- `GET /api/v3/servertime` — часы биржи в миллисекундах.

**Вежливое обращение.** Публичный API — общий ресурс. Проект опрашивает биржу по щадящему расписанию
(раз в 2 минуты), отправляет понятный идентифицирующий `User-Agent`, берёт все рынки одним запросом,
а не запрашивает каждый символ по отдельности, и обращается только к публичным endpoint'ам. Всегда
читайте Terms of Service
API, прежде чем писать автоматизацию поверх него.

**Serverless.** Вместо аренды машины, работающей круглосуточно, мы загружаем один **Worker**.
Cloudflare запускает его по трём расписаниям (сбор, агрегация, обслуживание) и при каждом
HTTP-запросе.

## Как это работает

```
cron */2 (каждые 2 мин)  ─▶  collector  ──GET──▶  Bitkub /market/ticker   (все рынки, 1 запрос)
                              │  привязать временной «bucket» (округлённую отметку) к часам Bitkub
                              │  проверить каждую запись · перевести цены в целые «минорные единицы»
                              └─▶ D1 (одна атомарная запись) + KV-кэш + метрики
cron ежечасно            ─▶  свернуть снимки в свечи OHLC за 1 час, затем за 1 день
cron ежедневно           ─▶  обновить каталог рынков · удалить данные за пределами окна хранения
HTTP-запрос              ─▶  read-API: /health · /v1/symbols · /v1/tickers/latest · …/:symbol · …/rollups
```

Это один Worker с тремя задачами по расписанию и HTTP-обработчиком, разбитый на маленькие части,
каждую из которых можно прочитать и протестировать отдельно.

### Read-API

| Маршрут | Что возвращает |
| --- | --- |
| `GET /health` | свежесть последних данных, число символов, счётчики аномалий |
| `GET /v1/symbols` | каталог отслеживаемых рынков |
| `GET /v1/tickers/latest` | последний снимок по каждому рынку |
| `GET /v1/tickers/:symbol?from=&to=&limit=` | исторические снимки по одному рынку |
| `GET /v1/tickers/:symbol/rollups?interval=1h\|1d` | свечи OHLC |

## Экскурсия по коду — пять уроков с привязкой к исходникам

Самые поучительные места кодовой базы. Каждое — урок, который переносится на другие проекты, и каждое
подсказывает, куда смотреть в файле.

1. **Целочисленная денежная арифметика** — [`src/domain/price.ts`](./src/domain/price.ts). Цены
   приходят строками (`"2017050.88"`). Float теряет точность на больших значениях, поэтому каждая
   цена становится **bigint в минорных единицах** и везде остаётся целым числом. *Смотрите:*
   `parseDecimalToMinor` (строка → bigint) и `formatMinorToDecimal` (обратно в строку); сначала
   прочитайте шапку файла.
2. **Знай пределы своей базы** — [`src/adapters/storage/d1.ts`](./src/adapters/storage/d1.ts). D1 не
   принимает `bigint` в bind и возвращает целочисленные колонки как JS-`number`, точный только до
   2⁵³−1. Поэтому значения проверяются по этой границе при разборе. *Смотрите:* короткий комментарий
   в шапке — он объясняет границу и почему нет защитной ветки. *Урок: измерьте, как ваше хранилище
   ведёт себя на самом деле, прежде чем проектировать вокруг него.*
3. **Идемпотентность** —
   [`src/adapters/storage/collect-store.ts`](./src/adapters/storage/collect-store.ts). Расписание
   иногда срабатывает дважды для одного bucket'а. Снимки пишутся через `INSERT OR IGNORE`, а строка
   журнала запуска имеет строгий уникальный ключ по bucket'у — поэтому второй запуск для того же
   bucket'а упирается в это ограничение и весь батч откатывается. Повторный запуск ничего не меняет.
   *Смотрите:* `commitCollect` и проверку `isUniqueConstraintError`.
4. **Арифметика времени без дрейфа** — [`src/config/cadence.ts`](./src/config/cadence.ts). Период
   сбора обязан делить 60 нацело (1, 2, 3, 4, 5, …). Значение вроде 7 или 13 рассогласует
   срабатывание cron и расчёт bucket'а на границе часа. *Смотрите:* `cronExprFor` (бросает исключение
   при неверном периоде — прямо при запуске Worker'а) и `bucketTsFor` (округляет отметку времени
   вниз до её bucket'а).
5. **Терпимость к плохим данным** — [`src/collector/collect.ts`](./src/collector/collect.ts) +
   [`src/adapters/bitkub/schemas.ts`](./src/adapters/bitkub/schemas.ts). Тикер проверяется по одной
   записи: сломанный или неизвестный рынок считается и пропускается, а остальные ~440 нормальных
   записей проходят. *Смотрите:* цикл по записям с `continue` в `collect.ts` и `safeParseTickerEntry`,
   возвращающий `null` для битой записи.

## Запустите сами

Два трека. **Локальный** не требует аккаунта Cloudflare. **Деплой** необязателен.

### Локальный трек (аккаунт не нужен)

```bash
git clone https://github.com/talkstream/thai-crypto-signals
cd thai-crypto-signals
pnpm install
pnpm cf-typegen                                                      # сгенерировать типы биндингов
pnpm exec wrangler d1 migrations apply thai-crypto-signals --local   # создать локальные таблицы
pnpm test:coverage                                                   # прогнать тесты
```

Тесты поднимают **настоящие локальные D1 и KV** — хранилище не подделано. Из заглушек — исходящий вызов к Bitkub (по записанным ответам) и продьюсер очереди: в локальном тест-рантайме нет API на стороне producer'а. Так понимается «тестирование на реальной инфраструктуре», и поэтому набор тестов ловит настоящие баги.

Теперь запустите Worker локально и вручную вызовите один тик сбора:

```bash
pnpm exec wrangler dev --test-scheduled                     # отдаёт http://localhost:8787
# во втором терминале — однократно запустить задачу сбора */2:
curl "http://localhost:8787/__scheduled?cron=*/2+*+*+*+*"
# затем прочитать данные, только что собранные из реального API Bitkub:
curl http://localhost:8787/health
curl http://localhost:8787/v1/tickers/latest
```

**✓ Контрольная точка.** `/health` должен выглядеть примерно так:

```json
{
  "ok": true,
  "nowMs": 1780772708158,
  "lastCollectBucketTs": 1780772640000,
  "lastCollectStatus": "partial",
  "lastObservedMs": 1780772647007,
  "symbolCount": 454,
  "recentDrift": 3,
  "recentScaleOverflow": 0
}
```

`ok: true` и `symbolCount` больше 0 означают, что всё сработало. `symbolCount: 0` почти всегда значит,
что пропущен шаг `/__scheduled` — база пуста, пока не пройдёт тик сбора. (`partial` и небольшой
`recentDrift` — это норма: несколько записей либо битые, либо не в каталоге, их пропускают; односторонние рынки сохраняются — с null в bid или ask.)

*Замечание:* локальный dev делает **настоящий** запрос к Bitkub с вашей машины, поэтому вызывайте
его нечасто.

**Если что-то пошло не так**
- `command not found: wrangler` → запускайте через `pnpm exec wrangler …`; Wrangler — зависимость
  проекта, а не глобальная программа.
- ошибка миграций → третье слово в команде должно быть ровно `thai-crypto-signals`, как имя в
  `wrangler.jsonc`.
- пустой `/health` → сначала дёрните `/__scheduled` (см. выше), потом читайте снова.

### Трек с деплоем (свой аккаунт Cloudflare, необязательно)

```bash
pnpm exec wrangler login
pnpm exec wrangler d1 create my-crypto-db
pnpm exec wrangler kv namespace create CACHE
pnpm exec wrangler queues create my-signals          # плюс очередь my-signals-dlq для dead-letter
# в wrangler.jsonc: подставьте свои database_id / id KV / имена очередей вместо исходных
pnpm exec wrangler d1 migrations apply my-crypto-db --remote
pnpm deploy
```

**✓ Контрольная точка.** Откройте `https://<ваш-worker>.workers.dev/health` — форма ответа та же, что
в локальной контрольной точке. Дайте cron `*/2` один тик (≈2 минуты), чтобы наполнить данные. Страница
ошибки Worker обычно значит, что миграции не применены с `--remote` либо id биндинга в `wrangler.jsonc`
остался исходным, а не вашим.

**Стоимость.** Весь проект укладывается в **бесплатный** тариф Cloudflare: cron-триггеры, D1, KV и — с февраля 2026 — Queues (10 000 операций в день) входят в него. Платный тариф для этой нагрузки не нужен.

**Уборка (чтобы не оставить ресурсы работающими)**

```bash
pnpm exec wrangler delete                             # сам Worker
pnpm exec wrangler d1 delete my-crypto-db
pnpm exec wrangler kv namespace delete --namespace-id <id>
pnpm exec wrangler queues delete my-signals
pnpm exec wrangler queues delete my-signals-dlq
```

## Упражнения

Попробуйте сами. У каждого упражнения — сложность и подсказка.

1. **(Просто)** Добавьте к `GET /v1/symbols` фильтр `?quote=THB`, чтобы возвращались только рынки в
   заданной котировке. *Подсказка:* фильтруйте каталог в `src/api/router.ts` до сериализации.
2. **(Средне)** Добавьте 15-минутный интервал свёрток рядом с 1h/1d. *Подсказка:* свёртка — это
   set-based SQL в `src/collector/rollup-job.ts`; помните про правило делителя 60 при выборе окна.
3. **(Средне)** Напишите тест на новый режим отказа — например, тикер вернул HTTP 500. *Подсказка:*
   единственная заглушка — `globalThis.fetch`; образец в `test/integration/collect.test.ts`.
4. **(Продвинуто)** Включите выключенный каркас **сигналов** и отправьте одно сообщение в Telegram при
   движении цены больше чем на X%. *Подсказка:* `src/signals/` есть, но пока не подключён к пути сбора, и `SIGNALS_ENABLED` в рантайме никто не читает — подключить продьюсер (и завязать его на этот флаг) и есть упражнение. Секреты кладут через `pnpm exec wrangler secret put …`. Делайте вежливо и с rate-limit.
